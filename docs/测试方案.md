# 测试方案

> 本文是 [架构重构方案](./架构重构方案.md) 的延伸讨论:重构成多包后测试如何组织。多包工程结构见 [多包工程组织方案](./多包工程组织方案.md)。

## 一、结论

**全仓统一用 vitest 一个运行器,各包配不同的测试环境和配套库;core 单独设立纯逻辑单元测试。**

需要先纠正一个常见误解:vitest 不是 Vue 专属的,它是通用测试运行器,React/Taro 一样可以用(Vite + React 项目的默认选择就是 vitest)。所以"适配不同框架就使用不同测试框架"不成立——运行器统一,变化的只是环境(node / happy-dom)和配套库(@vue/test-utils / @testing-library/react)。

## 二、各包的测试形态

| 包 | 运行器 | 环境 | 配套 |
| --- | --- | --- | --- |
| `core` | vitest | `node`(纯 JS,不需要 DOM) | 无需任何框架,手写 fake `ConditionSource` / `LifecycleWindow` |
| `uni` | vitest | `happy-dom` | `@vue/test-utils` + mock `@dcloudio/uni-app` 的生命周期 |
| `taro-react` | vitest | `happy-dom` / `jsdom` | `@testing-library/react` + mock `@tarojs/taro` 的 `useDidShow` 等 |

统一运行器的收益:一份断言 API、一份 mock 语法、根目录 `vitest` 一条命令全跑、覆盖率报告合并。vitest 支持 workspace/projects 模式,每个包一个 `vitest.config.ts` 声明自己的环境即可。

## 三、core 单测是分层架构的最大红利

现在的测试(`test/index.test.ts`)必须 mount 一个 Vue 组件、装 Pinia、mock uni-app 才能测到"条件等待"逻辑;分层后 core 的测试完全不需要这些:

```ts
// packages/core/test/whenAll.test.ts —— 零框架依赖
function fakeSource(initial: boolean) {
  let value = initial
  const listeners = new Set<() => void>()
  return {
    get: () => value,
    subscribe: (fn: () => void) => (listeners.add(fn), () => listeners.delete(fn)),
    set(v: boolean) { value = v; listeners.forEach(fn => fn()) }
  }
}

it('全部条件满足才 resolve,窗口关闭则取消', async () => {
  const a = fakeSource(false), b = fakeSource(true)
  const ac = new AbortController()
  const p = whenAll([a, b], ac.signal)

  a.set(true)
  await expect(p).resolves.toBeUndefined()
})
```

[架构重构方案](./架构重构方案.md) 中列出的手写状态机竞态 bug(条件翻转导致 promise 永久挂起、取消与用户异常混淆),在这种纯逻辑测试里都能直接覆盖到——而组件级测试根本测不到它们。

适配器包的测试相应变薄:只需验证"uni 的 onShow 正确翻译成了窗口开启、reactive/pinia 正确翻译成了 ConditionSource",条件组合逻辑不必在每个适配器里重复测。

## 四、现有测试的 `setTimeout` 断言问题(实证修正)

现有测试把断言写在 `setTimeout(() => { expect(...) }, 0)` 里且没有 await(`test/index.test.ts:16、26、41、55`)。这种写法是否有效,经过实际验证(写一个必然失败的断言放进同样的结构里运行),结论比直觉更微妙:

### 4.1 有效的部分

`setTimeout` 里失败的断言**不会被静默吞掉**:vitest 有全局 unhandled error 捕获,定时器回调里抛出的 `AssertionError` 会被记为文件级 Error,整个进程 exit code 为 1,**CI 会红**。因此"这些测试完全没有保护作用"的说法不成立。

### 4.2 但"等效 await"也不成立

实测输出暴露了三个差异:

1. **测试本身显示 passed**:报告是 `Tests 2 passed`——必然失败的测试被标记为通过,失败以文件级 "Error" 出现,不是测试失败;
2. **错误归因到错的测试上**:vitest 报错信息把错误怪到了**下一个测试**头上(`The latest test that might've caused the error is "对照组..."`),因为定时器触发时原测试早已结束。真实文件里有十几个用例时,排查会指向完全无关的用例;
3. **只能捕获"断言抛错"这一种失败**:这套保护来自 vitest 的全局错误陷阱,捕获不了"回调根本没执行"——比如钩子有 bug 导致目标值永远不被设置、断言恰好在组件 unmount 后才跑,这类时序问题 `setTimeout(0)` 不设防。

### 4.3 修正后的结论

现有写法在"断言失败时让 CI 变红"这一点上**有效**,但失败报告会误导排查方向,且对时序类 bug(恰恰是这个库最核心的风险面)没有确定性。

建议改成:

```ts
// 方式一:waitFor 会重试直到超时,等待语义明确
await vi.waitFor(() => expect(wrapper.vm.customMountedLogin).toBe(true))

// 方式二:刷掉微任务后同步断言
await flushPromises()
expect(wrapper.vm.customMountedLogin).toBe(true)
```

成本是每处一行,换来失败归因正确 + 等待语义明确(不是赌 0ms 后状态恰好就绪)。重构前的测试加固依然建议做([架构重构方案](./架构重构方案.md) 实施路线第 1 步),理由是**归因和时序确定性**,而非"测试失效"。
