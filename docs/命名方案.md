# 命名方案

> 配合 [架构重构方案](./架构重构方案.md) 一起阅读。重构后是多包结构(core + 各框架适配器),需要一个 npm scope,正是改名的最佳时机。

## 一、为什么要改名

`custom-hooks-plus` 有两个问题:

1. **"custom hooks" 没有信息量** —— 在 Vue/React 语境里泛指一切自定义钩子,完全没有传达本库的独特能力;
2. **"plus" 说不清是相对谁的 plus**。

好名字应该围绕库的本质语义:**生命周期窗口 + 条件就绪 → 触发**,即 "on show, **when** ready"。命名应落在 when / ready / gate 这几个词上。

## 二、候选方案

以下候选的 npm 包名经查询均未被占用(2026-06 查询;org/scope 页面有反爬,发布前需用 `npm org` 命令再确认一次):

| 候选 | 包形态 | 理由 |
| --- | --- | --- |
| **`@when-hooks/*`(推荐)** | `@when-hooks/core`、`@when-hooks/uni`、`@when-hooks/taro-react` | "when" 精确命中条件语义,API 读起来自然:`onShowWhen(['token'], cb)`;hooks 保留与现名的连续性 |
| `@lifegate/*` | `@lifegate/core`、`@lifegate/uni` | "生命周期 + 门控"合成词,短、独特、好记,品牌感最强,但语义需要一句话解释 |
| `@hookgate/*` | `@hookgate/core` | 直白表达 "hook 被 gate 住直到条件满足",但 "gate" 在前端语境稍偏后端/灰度发布 |
| `uni-when` | 单包名 | 若暂不拆包、仍主打 uniapp 生态,检索友好(`uni-` 前缀是 uniapp 插件市场惯例),但会限制将来 Taro 侧的品牌 |

## 三、推荐:`@when-hooks`

同时满足三点:

1. **说清楚库是干什么的** —— 条件钩子;
2. **适合多包 scope** —— core/adapter 分层天然对应 `@when-hooks/core`、`@when-hooks/uni`、`@when-hooks/taro-react`;
3. **API 命名能顺势统一**。

### API 改名对照

改名的同时一并梳理 API,把条件参数前置(更符合阅读顺序,也解决现在 cb 在前、条件在后的别扭参数序):

| 现有 API | 新 API |
| --- | --- |
| `onCustomShow(cb, keys)` | `onShowWhen(keys, cb)` |
| `onCustomLoad(cb, keys)` | `onLoadWhen(keys, cb)` |
| `onCustomMounted(cb, keys)` | `onMountedWhen(keys, cb)` |
| `onCustomLaunch(cb, keys)` | `onLaunchWhen(keys, cb)` |
| `onCustomReady(cb, keys)` | `onReadyWhen(keys, cb)` |
| `onCustomCreated(cb, keys)` | `onCreatedWhen(keys, cb)` |

## 四、迁移步骤

1. GitHub 仓库直接 rename(GitHub 会自动对旧地址做重定向);
2. 旧包 `custom-hooks-plus` 不删除,发一个最终版本,README 标注 deprecated 并指向新包;
3. 执行 `npm deprecate custom-hooks-plus "请迁移至 @when-hooks/uni"`;
4. 新包发布前用 `npm org` 确认 scope 可用。
