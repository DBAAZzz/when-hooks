# custom-hooks-plus 架构重构方案

> 本文档基于对 `core/init.ts`、`core/hooks.ts`、`core/rewrite.ts`、`utils/index.ts` 的代码评审,
> 给出现有架构的问题分析与重构方案,目标是:修复现有缺陷、支持跨框架(UniApp/Vue → Taro/React)扩展。

## 一、背景与结论

本库的核心思想——**"状态条件 × 生命周期时机"的交叉触发**(例如"token 就绪后再在 onShow 里执行")——是 UniApp 开发中的真实痛点,思想本身成立。

但当前实现存在两个层面的问题:

1. **实现路线问题**:在 Vue 3 内部手写了一套响应式系统(Proxy + Promise 状态机),约 60% 的代码在重新发明 `reactive` + `watch`,并引入了 Vue 原生方案不会有的 bug。
2. **架构分层问题**:框架耦合点散落各处(`utils` 依赖 vue 的 watch、`rewrite` 依赖 uni-app 生命周期),没有把"条件等待语义"和"框架绑定"分开,导致无法移植到 Taro 等其他框架。

**结论:做 core/adapter 分层。内核是零框架依赖的纯 JS,面向最小订阅契约编程;各框架(UniApp/Vue、Taro/React)只写百行级的适配器。**

## 二、现有架构的问题清单

### 2.1 架构性问题

#### 问题 1:手写 Proxy 只能监听顶层属性,嵌套赋值丢失

`createProxy` 的 set trap 只拦截顶层属性赋值(`core/init.ts:82`):

```ts
globalData.userInfo = { name: 'x', ... }  // ✅ 触发 track
globalData.userInfo.name = 'x'            // ❌ 不触发,set trap 不会执行
```

`track()` 里的 `flattenObject` 扁平化只在"整个对象被替换"时才有机会运行,而用户最自然的写法恰恰是直接修改嵌套属性。配置了 `userInfo.name` 的监听却要求用户必须整对象替换才能触发,是隐蔽的功能性缺陷。

#### 问题 2:Promise 状态机存在"永久挂起"窗口

条件从满足翻转为不满足时,`track` 会创建新的 pending promise 并**覆盖** `promiseMap` 中的 entry(`core/init.ts:53`),旧 promise 的 `resolve` 随旧 entry 丢失。此时若有 hook 正在 `await Promise.all(iterable)`(它持有的是快照时的旧 promise),该 promise 永远不会 resolve;后续条件再次满足时 resolve 的是新 promise。等待中的 hook 静默失联,只能靠 `onHide`/`onUnload` 的 reject 解围。

同时,被替换的旧 promise 留在 `promiseCache` 中没有清理路径,构成缓慢的内存泄漏。

#### 问题 3:全局单例 + 隐式时序耦合

`customHooks` 是模块级单例(`core/init.ts:132`):

- `createProxy` 会整体覆盖 `this.proxyTarget`,全应用只能有一个 global 代理对象,第二次调用悄悄顶掉第一个;
- `init()` 读取 `this.proxyTarget` 取默认值,因此**必须先 createProxy 再 init**,但该时序约束无任何校验,违反时只得到 undefined 初值;
- `init()` 可被重复调用,每次都对 pinia store 重新挂 `watch`,而 `watchs()` 返回的 stop handle 被直接丢弃(`utils/index.ts:80`),无任何卸载/dispose 机制;
- 单例导致单元测试无法隔离、SSR 下跨请求串状态。

#### 问题 4:用字符串拼接编码结构化信息,解析脆弱

pinia key 被编码为 `[pinia|storeId]-key`,而 `restoreKey` 用 `key.split("-")[1]` 还原(`utils/index.ts:70`)。只要 store id 或属性名本身含 `-`(如 `user-info.name`),解析即出错。

此外 `init()` 中 `w.key = key` 直接改写了**用户传入的配置对象**(`Object.assign({}, ...)` 是浅拷贝,`w` 仍是用户的引用),属于副作用外泄。

### 2.2 次级问题

| 问题 | 位置 | 说明 |
| --- | --- | --- |
| 异常被整体吞掉 | `core/hooks.ts:79` | `catch (e) {}` 把"取消"和"用户回调抛错"混在一起静默,用户 cb 里的 bug 无声消失 |
| `_onCreated` 语义不实 | `core/rewrite.ts:48` | 注册时同步执行回调,并非真正的 created 时机,与其他钩子语义不一致 |
| 同引用对象误判"没变" | `core/init.ts:84` | `oldValue === newValue` 短路对同引用对象的内部变更失效,与问题 1 叠加 |
| `flattenObject` 无防护 | `utils/index.ts:14` | 循环引用栈溢出;数组逐索引展开,key 数量可能爆炸 |
| 类型系统未兑现价值 | `types/index.ts` | `store: any`、`onUpdate: (val: any)`、`watchKey: string \| string[]`;应让 watchKey 收窄为已注册 key 的字面量联合,拼错 key 编译期报错 |
| 工具链与包元数据 | `package.json` | TS 4.9 / Rollup 3 较老;缺少 `exports`/`module` 字段,ESM 产物元数据不完整 |

## 三、关于"跨框架"的两个关键澄清

### 3.1 现在的代码本来就跑不到 Taro 上

手写 Proxy 并没有买到框架无关性,Vue 依赖早已渗透进核心:

- `utils/index.ts:1` 直接 `import { watch } from "vue"`——pinia 模式整个建立在 Vue watch 上;
- `core/rewrite.ts:1-2` 硬依赖 `@dcloudio/uni-app` 与 Vue 生命周期;
- peerDependencies 就是 vue + uni-app。

框架耦合点根本不在 `createProxy` 这一处。要支持 Taro,无论内核用什么方案都得重构;手写 Proxy 在可移植性上没有提供任何保护。

### 3.2 `@vue/reactivity` 本身是框架无关的

`reactive`/`effect`/`computed` 发布在独立包 `@vue/reactivity` 中,**不依赖 Vue 运行时**,可在 React、Node、任何环境使用。Taro 的 React 项目里 `import { reactive, effect } from '@vue/reactivity'` 完全合法。

因此"用 reactive 替换手写 Proxy"与"将来支持 Taro"并不冲突——前提是依赖 `@vue/reactivity` 而非 `vue`。

## 四、目标架构:core/adapter 分层

### 4.1 核心洞察

**"等待若干条件全部满足"本质上是 watch 一个 computed boolean,而不是管理一组 promise 的生命周期。**

整个 `promiseCache`/`promiseMap`/`track`/`flattenObject`/`findCommonSubset` 体系可以收缩为"条件即布尔、等待即 watch、取消即 AbortSignal"。

跨框架移植真正要处理的只有三个变化点,它们都应该是**适配器**,不是内核:

| 变化点 | UniApp/Vue | Taro/React |
| --- | --- | --- |
| 生命周期窗口 | `onShow` / `onHide` / `onLoad` | `useDidShow` / `useDidHide` / `useLoad` |
| 状态源 | reactive 对象、Pinia | zustand / redux / mobx、普通对象 |
| 注册时机 | setup 中调用 | 函数组件渲染中调用 |

### 4.2 包结构

```text
@when-hooks/core         whenAll、生命周期窗口语义、取消、key 注册 —— 纯 JS,零框架依赖
@when-hooks/uni          onShow→window,reactive/pinia→ConditionSource
@when-hooks/taro-react   useDidShow→window,zustand.subscribe→ConditionSource
```

> 多包之间如何相互引用(开发期引源码、发布时 external)见 [多包工程组织方案](./多包工程组织方案.md);包命名的由来见 [命名方案](./命名方案.md)。

### 4.3 内核契约(零框架依赖)

内核面向一个最小订阅契约编程(与 React `useSyncExternalStore` 的契约同构):

```ts
/** 一个可订阅的布尔条件源 —— 各框架适配器负责把自己的状态翻译成它 */
interface ConditionSource {
  get(): boolean
  subscribe(onChange: () => void): () => void  // 返回 unsubscribe
}

/** 等待全部条件满足;窗口关闭时通过 signal 取消 */
function whenAll(sources: ConditionSource[], signal: AbortSignal): Promise<void>

/** 生命周期窗口 —— 适配器把 onShow / useDidShow 接进来 */
interface LifecycleWindow {
  onOpen(cb: () => void): void   // onShow / useDidShow
  onClose(cb: () => void): void  // onHide / useDidHide
}
```

`whenAll` 的参考实现(纯 JS,无框架依赖):

```ts
function whenAll(sources: ConditionSource[], signal: AbortSignal): Promise<void> {
  return new Promise((resolve, reject) => {
    const unsubs: Array<() => void> = []
    const cleanup = () => unsubs.forEach(u => u())

    const check = () => {
      if (sources.every(s => s.get())) {
        cleanup()
        resolve()
      }
    }

    signal.addEventListener('abort', () => {
      cleanup()
      reject(signal.reason)
    })

    sources.forEach(s => unsubs.push(s.subscribe(check)))
    check() // 立即检查一次,条件已满足则同步 resolve
  })
}
```

要点:

- **条件翻转只是布尔值变化**,不存在 promise 身份过期,问题 2 消失;
- **取消用 `AbortController`** 表达,`shared` 对象、uuid、`getSharedKey` 全部可删,且"取消"和"用户回调抛错"天然分离(只 catch AbortError,用户异常照常抛出),解决异常吞噬问题;
- key 注册用结构化对象 `{ storeId, path }` 表达,展示用字符串只做展示,`generatePiniaKey`/`restoreKey` 的字符串协议可删,问题 4 消失。

### 4.4 Vue/UniApp 适配器

各状态源到 `ConditionSource` 的翻译成本都很低:

```ts
// Vue reactive / Pinia → ConditionSource
function fromVueSource(getter: () => unknown, onUpdate?: (v: any) => boolean): ConditionSource {
  const cond = computed(() => {
    const val = getter()
    return onUpdate ? onUpdate(val) : !!val
  })
  return {
    get: () => cond.value,
    subscribe: (onChange) => watch(cond, onChange) // 返回的 stop 即 unsubscribe
  }
}

// UniApp 生命周期 → LifecycleWindow
const showWindow: LifecycleWindow = {
  onOpen: (cb) => onShow(cb),
  onClose: (cb) => { onHide(cb); onUnload(cb) }
}
```

`createProxy` 直接变成 `reactive()` 的薄封装:

- 嵌套属性赋值天然可监听,问题 1 消失;
- `watch` 返回的 stop handle 配合 `effectScope` / `onScopeDispose`,组件卸载自动清理,解决泄漏;
- Pinia 侧 `store.$subscribe` 本来就是订阅契约的原生形态。

### 4.5 Taro/React 适配器(未来)

- `useDidShow` / `useDidHide` 与 uni 的 onShow/onHide 一一对应,翻译为 `LifecycleWindow`;
- zustand 的 `store.subscribe` 原生就是 `ConditionSource` 契约;
- "可变全局代理对象"(createProxy)这一范式在 React 生态并不习惯,**不建议强行移植**。两个选择:
  - core 里基于 `@vue/reactivity` 提供(两端通用,深层监听白送);
  - 只在 Vue 适配器里提供,Taro 侧引导用户使用 zustand 等其生态原生的状态源。

### 4.6 实例化:工厂代替单例

用 `createCustomHooks()` 工厂返回实例(类似 `createPinia()`),解决问题 3:

```ts
const hooks = createCustomHooks()
hooks.register({ ... })          // 代替全局 init
```

导出一个默认实例以兼容现有 `init`/`createProxy`/`onCustomShow` API,旧用户无感。

## 五、保留并强化的部分

`rewrite.ts` 里"生命周期窗口"的抽象(onShow 开启等待、onHide/onUnload 取消)是本库**真正独有的价值**——Vue/UniApp 生态没有现成答案。应保留并强化为显式概念:

```ts
createLifecycleWindow(startHook, endHooks)
```

条件等待这一半交给响应式系统/订阅契约去做。预计内核代码量减少一半以上,同时消除第二节列出的大部分 bug 面。
