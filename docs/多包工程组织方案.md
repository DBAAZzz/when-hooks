# 多包工程组织方案:uni 如何引入 core

> 本文是 [架构重构方案](./架构重构方案.md) 的延伸讨论:重构成 `core`、`uni`、`taro-react` 多包后,适配器包如何引入 core 的代码——引产物还是引源码,主流方案是怎样的。包命名见 [命名方案](./命名方案.md)。

## 一、结论

**主流方案是"开发期引用源码、发布产物里 core 是外部依赖(不打包进去)"**,两件事分别处理:

1. **发布形态**:core 是 uni 的运行时依赖(external),不内联进产物;
2. **开发期解析**:通过 `workspace:^` + `publishConfig` 机制,开发期直接解析到 TS 源码,发布时自动切回 dist 产物。

## 二、发布形态:core 是 uni 的运行时依赖,不内联

`@when-hooks/uni` 的 `package.json` 这样声明:

```jsonc
// packages/uni/package.json
{
  "name": "@when-hooks/uni",
  "dependencies": {
    "@when-hooks/core": "workspace:^"   // pnpm workspace 协议
  }
}
```

要点:

- **`workspace:^` 协议**:开发期 pnpm 把它链接到本地 `packages/core` 目录;执行 `pnpm publish` 时自动替换为真实版本号(如 `^0.1.0`)。这是 pnpm monorepo 的标准做法,Vue、VueUse、Element Plus、Pinia 都是这么做的。
- **构建时 external,不打包**:uni 包的构建配置把 `@when-hooks/core` 标记为 external,产物里只留 `import { whenAll } from '@when-hooks/core'` 这行 import,运行时由用户的包管理器装好 core 来满足。

```js
// packages/uni/rollup.config.mjs
export default {
  external: ['@when-hooks/core', 'vue', '@dcloudio/uni-app'],
  // ...
}
```

### 为什么不把 core 打包进每个适配器(内联)?

如果用户同时装了 uni 和 taro 适配器(或者 core 修了 bug 只升 core),内联会导致出现两份不同版本的 core 副本,单例状态、`instanceof` 判断都会出问题。external + 真实依赖让包管理器去解决版本归一,这是库生态的默认正确做法。内联只在 core 极小且确定永远不会被单独依赖时才考虑。

## 三、开发期:uni 怎么"看到" core 的代码

这里才是"产物还是源码"分歧的真正所在,主流有两派:

### 方案 A:引产物 + watch 构建

每个包各自构建出 dist,uni 通过 core 的 `exports` 字段解析到 `dist/index.js`。开发时跑 `turbo watch build` 或各包 `tsup --watch`,core 一改自动重新构建。

- **优点**:开发期看到的就是用户将来拿到的东西,不会出现"源码下能跑、发布后坏了"的偏差。
- **缺点**:改 core 要等一轮构建;IDE 跳转定义落到 `.d.ts` 而不是源码,开发体验差一截。
- **代表**:Turborepo 官方文档里的 "compiled packages" 模式。

### 方案 B:开发期直接解析到源码,发布时切回 dist(主流库的选择)

core 的 `package.json` 开发期指向源码,发布时由 `publishConfig` 覆盖为产物路径——**pnpm 在 publish 时会用 `publishConfig` 里的字段替换顶层字段**:

```jsonc
// packages/core/package.json
{
  "name": "@when-hooks/core",
  "type": "module",
  "main": "./src/index.ts",          // 开发期:直接指到 TS 源码
  "exports": { ".": "./src/index.ts" },
  "publishConfig": {                  // 发布时自动覆盖 ↓
    "main": "./dist/index.js",
    "types": "./dist/index.d.ts",
    "exports": {
      ".": { "types": "./dist/index.d.ts", "import": "./dist/index.js" }
    }
  }
}
```

- **优点**:改 core 即时生效、零等待;IDE 跳转直达源码;测试(vitest)天然跑在源码上,断点、覆盖率都直接。
- **缺点**:开发期消费的是 TS 源码,要求 uni 包的构建工具/测试工具能编译它(rollup + esbuild/swc、vitest 都没问题);必须有 CI 步骤在发布前验证构建产物本身可用,否则"源码能跑、产物坏了"会漏到线上。
- **代表**:VueUse、unjs 全家桶(unbuild 的 stub 模式本质就是这个思路)、大量现代 TS monorepo。

还有一个变体:不用 `publishConfig`,而是在 vitest/tsconfig 里加 alias 把 `@when-hooks/core` 指到 `packages/core/src`。效果类似,但配置分散在多处,不如 `publishConfig` 一处声明干净。

## 四、对本项目的建议

包就 2~3 个、core 很小,建议 **pnpm workspace + 方案 B**:

```text
when-hooks/
├── pnpm-workspace.yaml        # packages: ['packages/*']
├── packages/
│   ├── core/                  # 纯 JS,产物 ESM + d.ts
│   ├── uni/                   # dependencies: @when-hooks/core: workspace:^
│   └── taro-react/            # 同上
```

配套三件事:

1. **构建工具换成 tsup 或 unbuild**——一条命令出 ESM + d.ts,三个包共享一份配置。现有仓库里的 rollup + babel + api-extractor 三步流程(`build-bundle` / `build-types` / `build-api-types`)可以整个删掉;
2. **测试放各包内**,vitest workspace 模式一次全跑。因为方案 B 的 import 直达源码,跨包断点调试开箱即用;
3. **CI 里加一步 `pnpm -r build && pnpm -r publint`**(或 attw),发布前验证产物和 `exports` 字段没坏,补上方案 B 唯一的风险点。

## 五、一句话总结

**依赖关系上 uni 永远"引 core 这个包"(external,不内联);解析目标开发期是源码、发布后是产物,靠 `workspace:^` + `publishConfig` 两个机制自动切换——这就是当前 TS 库 monorepo 的主流形态。**
